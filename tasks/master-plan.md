# Dream Store — Master Plan

> 사내 AI 도구 마켓플레이스 · 컨셉 시연용 목업
> 작성일: 2026-07-22 | 시연: 2026-07-23 09:00

---

## 1. 서비스 개요

**드림 스토어(Dream Store)** 는 임직원이 직접 만들거나 발굴한 AI 도구(Skill / MCP / 프롬프트)를
등록하고, 다른 임직원이 설치해 사용하고, 평가하는 **사내 AI 도구 마켓플레이스**다.

> **핵심 비유** — AI 에이전트는 스마트폰이다.
> App Store에서 앱을 설치해 스마트폰이 유용해지듯,
> 드림 스토어에서 도구를 설치해 나의 AI 에이전트가 강력해진다.
> 그리고 앱 개발자가 다운로드 수와 별점으로 평가받듯,
> 도구를 만든 임직원도 같은 방식으로 평가받고 **보상받는다.**

## 2. 배경 및 문제 정의

**배경** — 회장님 지시: *"AI를 잘 쓰는 임직원에게 활용 역량에 따라 성과금 또는 상품을 수여하고 싶다."*

이를 실행하려면 두 가지가 먼저 풀려야 한다.

| 문제 | 현재 상태 | 드림 스토어의 해법 |
| --- | --- | --- |
| **역량을 측정할 방법이 없다** | "누가 AI를 잘 쓰는가"를 판단할 객관적 근거 부재 | 설치 수·재사용률·평점이라는 **동료 검증 기반 지표**로 대체 |
| **노하우가 개인에 갇혀 있다** | 잘 쓰는 사람의 방법이 그 사람 안에서만 소비됨 | 도구 형태로 **패키징 → 유통 → 전사 확산** |

즉 드림 스토어는 시상을 위한 **측정 도구**인 동시에, 그 과정 자체가 **전사 AI 역량 상향 평준화 엔진**이다.

## 3. 타겟 사용자

**모수: 사내 AI 사용 임직원 약 200명** (Claude 약 140명 / ChatGPT 약 60명)

세 개의 역할이 순환하는 구조로 설계한다.

| 페르소나 | 하는 일 | 얻는 것 |
| --- | --- | --- |
| **빌더 (Builder)** | Skill·MCP를 직접 제작해 등록 | 차별화된 AI 응용 역량 증명 → **창작자 시상** |
| **스카우트 (Scout)** | 검증된 오픈소스 MCP를 발굴·큐레이션해 등록 | 개발 역량 없이도 기여 가능 → **발굴자 시상** |
| **유저 (User)** | 설치해 쓰고 구조화된 후기 작성 | 업무 효율 즉시 개선 · 평가 참여로 기여 |

> **설계 의도** — 스카우트 트랙이 있기에 비개발 직군(영업·인사·재무·기획)도 첫날부터 참여할 수 있다.
> Skill은 마크다운 문서로 작성되므로, 빌더 역시 개발자만의 영역이 아니다.
> 이는 회장님 지시의 취지인 *"AI 활용 역량"* — 코딩 실력이 아닌 **응용 능력** — 에 정확히 부합한다.

## 4. 핵심 가치 제안

- **만드는 사람** — 자신의 AI 활용 노하우를 도구로 만들어 공개하고, *"이 업무에 이렇게 써서 효율이 이만큼 좋아졌다"* 를 사례로 공유한다. 남과 차별화된 AI 응용력이 **측정 가능한 형태로 증명**된다.
- **쓰는 사람** — 잘하는 동료의 방법을 설치 한 번으로 그대로 흡수한다. 개인차가 줄고 **조직 전체가 상향 평준화**된다.
- **회사** — 전사 AI 활용 수준이 **절감 시간이라는 숫자**로 집계되고, 시상 대상자를 납득 가능한 근거로 선정할 수 있다.

## 5. 등록 대상 도구

### 5.1 등록 유형

| 구분 | 내용 | 시상 트랙 |
| --- | --- | --- |
| **자체 제작** | 본인이 만든 Skill / MCP / Subagent / 프롬프트 템플릿 | 빌더 |
| **큐레이션** | 직접 써보고 검증한 **오픈소스 MCP·Skill 추천 등록** | 스카우트 |

> 큐레이션 등록을 허용함으로써 오픈 첫날부터 스토어가 채워지고(콜드 스타트 해소),
> "좋은 도구를 알아보는 눈" 자체를 하나의 역량으로 인정한다.
> 단, 큐레이션 등록에는 **실사용 후기와 사내 적용 사례 기재를 필수**로 하여 단순 링크 수집과 구분한다.

### 5.2 등록 메타데이터

| 항목 | 설명 |
| --- | --- |
| 도구명 / 아이콘 | — |
| 유형 | Skill · MCP · Subagent · 프롬프트 |
| 등록 구분 | 자체 제작 · 큐레이션 |
| **대상 에이전트** | Claude · ChatGPT · 공통 |
| **직무 카테고리** | 영업 · 인사 · 재무 · 기획 · 개발 · 마케팅 · 공통 |
| 한 줄 소개 / 상세 설명 | — |
| **사용 사례** | 어떤 업무에 어떻게 썼는지 (필수) |
| 설치 가이드 | 원클릭 복사용 설정 스니펫 |
| 등록자 / 소속 부서 | — |

> **카테고리를 기술 기준(MCP/Skill)이 아니라 직무 기준으로 노출**하는 것이 핵심이다.
> 영업사원은 "MCP"를 찾지 않는다. "영업 업무에 쓸 만한 것"을 찾는다.

## 6. 평가 체계 — 3층 + 거버넌스

### 6.1 설계 원칙

> **상금이 걸리는 순간 별점은 화폐가 된다.**
> 사내는 모수가 작고 서로 아는 사이라 "우리 팀 것 좀 눌러줘"가 쉽게 발생한다.
> 따라서 단일 지표에 의존하지 않고, **복합 지표 + 거버넌스 심사**로 방어한다.
> 거버넌스 위원회의 적합성 평가가 최종 방어막 역할을 한다.

### 6.2 3층 지표

| 층 | 지표 | 조작 난이도 | 비중 |
| --- | --- | --- | --- |
| **① 사용 실적** | 설치 수, **재사용률**, 월간 활성 사용자 | 높음 (신뢰도 최상) | **40%** |
| **② 동료 평가** | 별점 + 구조화 후기 + 누적 절감 시간 | 낮음 | **30%** |
| **③ 거버넌스 심사** | 운영위원회 적합성·독창성·확산성 평가 | — | **30%** |

> **메인 지표는 "설치 수"가 아니라 "재사용률"이다.**
> 부탁해서 받은 설치는 재사용률에서 걸러진다. 한 번 깔고 안 쓰는 도구는 점수가 오르지 않는다.

### 6.3 거버넌스 위원회

- **구성** — AI 추진 담당 부서 + 각 본부 대표 (분기 단위 운영)
- **역할**
  - 지표 상위 후보군에 대한 **정성 심사**(독창성 / 업무 기여도 / 확산 가능성)
  - 이상 패턴(단기간 집중 설치, 동일 부서 편중 평가) **검토 및 조정**
  - 자체 제작 vs 큐레이션 트랙별 균형 조정
- **의의** — "왜 저 사람이 받았느냐"는 사후 논란을 사전 차단하는 장치

### 6.4 구조화 후기 양식

자유 서술만 받으면 데이터가 남지 않는다. 후기는 아래 항목으로 구조화한다.

| 항목 | 입력 방식 |
| --- | --- |
| 별점 | 1~5 |
| 어떤 업무에 사용했는가 | 드롭다운 (문서작성 / 데이터분석 / 코드 / 고객대응 / 회의·보고 / 기타) |
| 도입 전 소요 시간 | 숫자 (분/회) |
| 도입 후 소요 시간 | 숫자 (분/회) |
| 월 사용 빈도 | 숫자 (회/월) |
| 한 줄 사례 | 텍스트 |

→ **월 절감 시간 = (도입 전 − 도입 후) × 월 빈도** 로 자동 계산되어
개인·도구·전사 단위로 집계된다.

> 이 집계의 최종 산출물이 **"드림 스토어 도입 후 전사 월 ○○○시간 절감"** 이라는 단일 숫자다.
> 경영 보고에서 이 숫자 하나가 나머지 지표 전부보다 강하다. 랜딩 페이지 최상단에 배치한다.

## 7. 시상 체계 *(초안 — 채택 후 확정)*

| 트랙 | 대상 | 선정 기준 |
| --- | --- | --- |
| **최우수 빌더** | 자체 제작 도구 종합점수 1위 | 3층 종합 점수 |
| **베스트 스카우트** | 큐레이션 등록 종합점수 1위 | 3층 종합 점수 |
| **확산상** | 가장 많은 동료가 실사용한 도구 | 재사용률 × 활성 사용자 |
| **효율왕** | 누적 절감 시간 1위 도구 | 구조화 후기 집계 |
| **베스트 리뷰어** | 양질의 후기를 가장 많이 남긴 사용자 | 후기 수 × 도움됨 수 |

- **주기(분기/반기)와 상금 규모는 아이디어 채택 후 확정한다.**
- 리뷰어 시상을 두는 이유: 평가 데이터의 양과 질이 시스템 전체의 신뢰도를 결정하기 때문.

## 9. 기술 스택 및 배포

| 영역 | 선택 |
| --- | --- |
| 호스팅 | **GitHub Pages** (`hyeonseokhan/Dream-Store`) |
| 백엔드 | **Supabase** (DB · 인증 대용) |
| 프론트 | 정적 SPA |
| 시드 데이터 | 오픈소스 MCP 기반 실존 도구 + 사내 가상 도구 조합 |

> **시연 안정성 우선 원칙** — 화면과 시드 데이터로 전체 흐름이 완결되도록 먼저 구성하고,
> Supabase 연동은 **후기 작성·설치 카운트** 등 "실제로 동작하는 장면"에 한정해 얹는다.
> 네트워크 문제로 시연이 멈추는 상황을 원천 차단한다.

## 10. 시연 시나리오 (5분)

1. **랜딩** — "전사 월 1,240시간 절감" 카운터로 시작. 규모감을 먼저 각인.
2. **스토어 탐색** — 직무 필터에서 *영업* 선택 → 영업 담당자 관점으로 도구 탐색.
3. **도구 상세** — 사용 사례와 동료 후기 확인 → **설치 버튼 클릭** (설정 스니펫 복사).
4. **후기 작성** — 도입 전/후 시간 입력 → **절감 시간이 즉시 계산되어 전사 카운터에 반영**되는 장면.
5. **시상 대시보드** — 랭킹과 종합점수 구성 공개 → *"이 근거로 이번 분기 수상자를 선정합니다"* 로 마무리.

## 11. 향후 과제 (이번 범위 제외)

- **보안·적정성 심사 파이프라인** — MCP는 본질적으로 외부 연결을 수반하므로, 업로드 → 심사 → 승인 → 게시 게이트와 접근 범위 배지(사내 전용 / 외부 API / 파일 접근)가 필요. **아이디어 채택 후 정식 설계 시 반드시 반영.**
- 그룹웨어 SSO 연동 및 사번 기반 인증
- 실제 설치 여부 자동 검증 (에이전트 텔레메트리 연동)
- 시상 주기 및 상금 규모 확정
